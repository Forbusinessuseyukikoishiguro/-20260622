# うさうさ現場あんしん工房 v2.8 ｜ 設計書【新人エンジニア向け・完全版】

> この設計書は、ツールを「ゼロから完璧に再現できる」ことを目的に、やさしい言葉で書いています。
> 専門用語には初出でかんたんな説明をつけます。難しく感じたら、まず「処理流れ」の章だけ読めばOKです。

---

## 0. このツールは何？（1分でわかる）

PoC（試行錯誤しながら小さく作って検証する案件）の現場で、**お客さまと後で揉めないためのコミュニケーションを助ける道具箱**です。
1つのHTMLファイルだけで動き、ネットが無くても多くの機能が使えます。AI相談だけはネット接続が必要です。

たとえるなら「うさうさラーメン店🍜」。注文（依頼）を受けて、味の好み（ゴール）を確かめ、作る順番を決め、味見（テスト）をして、最後に「この味で合ってますか？」と確認する——その一連を、画面の中で型として用意したものです。

---

## 1. 要件定義（何を作るか）

### 1-1. 解きたい課題
- 依頼内容の認識ズレ、スコープのなしくずし、「言った／言わない」問題で、PoCはよく揉める。
- 多忙でドキュメントを書く余裕がない。

### 1-2. 機能要件（やること）
1. フェーズごとの勘所・心理的原則・危険サインを引ける（①）
2. 「受けて→復唱→認識を返す→行動宣言」を記録できる（②）
3. きつい言い方を和風にやわらげる（③）
4. 合意メモを作って書き出す（④）
5. AIにやさしく壁打ちできる・音声対応（⑤⑧⑨⑩）
6. タスク分割と報連相（⑥）
7. 初日の進め方テンプレ（⑦）
8. 依頼書の読み合わせ／無いときの叩き台作成（⑩）
9. 学習ノート・資格ロードマップ（⑨）

### 1-3. 非機能要件（やり方の条件）
- **オフラインファースト**：AI以外は通信なしで動く。
- **単一HTML**：ビルド不要。ダブルクリックで開ける。
- **データは端末内**：勝手に外部送信しない（AI送信時のみ明示）。
- **アクセシビリティ**：キーボード操作・読み上げに配慮（aria属性）。
- **嘘をつかない**：AIに出典を捏造させない。

## 2. 設計の方針（なぜこう作るか）

| 方針 | 理由（新人向けの一言） |
|------|------------------------|
| 単一HTML・素のJS | 環境構築でつまずかない。配って即動く。 |
| 状態は1つのオブジェクト | 「いまの状態」をまとめて保存・復元できる。 |
| 保存先を自動判定 | claude.ai上でもローカルでも同じファイルで動かすため。 |
| 機能ごとにモジュール化 | 1機能=1かたまり。読みやすく直しやすい。 |
| 画面は「データ→描画」 | データを変えたら描き直す、の繰り返し。バグりにくい。 |
| AIは“足し算” | 無くても基本機能は動く。AIは便利さの上乗せ。 |

## 3. 詳細設計（中身のしくみ）

### 3-1. 全体構成
- **同意ゲート**：起動時に「データの行き先」を説明し、同意（初回必須・保存される）。
- **アプリ本体**：10個のタブ（①〜⑩）。
- **横断機能**：保存／バージョン／インポート・エクスポート、扇子UI（使い方）、琴の調律（AIパラメーター）。

### 3-2. 保存のしくみ（Store）
保存先を次の順で自動的に選びます。
1. `window.storage`（claude.aiのアーティファクト保存）
2. `localStorage`（ダウンロードしたHTMLをブラウザで開いたとき）
3. メモリのみ（どちらも使えない環境。エクスポートで持ち出す）

```
get / set / del / listKeys
```
の4つを async（待てる関数）で包み、どの保存先でも同じ書き方で使えるようにしています。

### 3-3. 状態（snapshot）
画面の入力やログを1つのオブジェクトにまとめます（これを「スナップショット」と呼びます）。
- 例：`log`（指示受けログ）、`tasks`（タスク）、`kick`（初日テンプレ）、`ume`/`conc`/`saku`（各AI会話）、`learnNotes`、`koto`（調律）など。
- 「一時保存」はこのスナップショットを保存先に書き、起動時に読み戻します。

### 3-4. AI機能のしくみ
- ブラウザから `https://api.anthropic.com/v1/messages` に問い合わせます（claude.ai上で動く前提）。
- モデルは `claude-sonnet-4-6`。会話履歴を毎回まとめて送ります。
- 論文を参照するときは `web_search` ツールを一緒に渡し、回答に出典を書かせます。
- システムプロンプト（AIへの役割指示）に「嘘の出典は作らない」を必ず入れます。

### 3-5. 音声のしくみ
- ブラウザの音声認識（SpeechRecognition）を使います。
- 「確定した文字＝濃い墨」「認識中＝薄字」で色分けし、**どう聞き取られたか**を見せます。
- 使えない環境ではボタンを無効化し、テキスト入力に誘導します。

## 4. 実装（主要パーツ）

| パーツ | 役割 |
|--------|------|
| `Store` | 保存先の自動判定と読み書き |
| `attachSpeech()` | 梅/桜ボタンに音声認識を付ける（色分け付き） |
| `callClaude()` | AIへの問い合わせ（温度・長さ・ツールを反映） |
| `renderXxx()` | データから画面を描く関数群 |
| `snapshotState()` / `applyState()` | 状態の保存と復元 |
| `demoRecognition()` | 音声色分けの動作デモ（マイク不要） |

実装ルール（新人が守ると安全）：
- 文字は必ず `esc()` でエスケープしてから画面に入れる（タグ混入を防ぐ）。
- 画面を直接いじらず、データを変えて `render` し直す。
- 例外は `try/catch` で受け、ユーザーには優しいメッセージを出す。

## 5. 処理の流れ（ユーザー操作→内部）

### 5-1. 起動
1. 同意ゲート表示（初回）→ 同意を保存・履歴に記録。
2. 2回目以降は同意を読み取り、ゲートを自動スキップ。
3. アプリを組み立て、前回の保存（autosave）があれば復元。

### 5-2. 依頼書がないとき（⑩→④/⑦）
1. ⑩で項目を入力 →「叩き台を作る」。
2.「④確認メモへ」「⑦初日テンプレへ」で各タブに転記。
3. ④や⑦で仕上げてコピー／書き出し。

### 5-3. AI相談（⑤⑧⑨⑩）
1. 文字か音声で質問。
2.（論文参照ONなら）web_searchで裏取り→出典つきで回答。
3.（⑨⑩）回答を「②ログ」「⑥タスク」へ保存できる。

## 6. テスト（品質の守り方）

- **構文チェック**：`node --check` でJSの文法エラーを検出。
- **動作テスト**：`jsdom`（ブラウザの代わり）でHTMLを実行し、ボタン操作・保存・復元を自動確認。
- **AIはモック**：通信せず固定応答に差し替えて、ロジックだけ検証。
- v2.8 のテスト結果：**30項目すべて成功（0失敗）**。
- 限界：実マイクの音声認識は自動テストでは確認不可 → 「音声色分けデモ」で見え方を確認。

## 8. 用語ミニ辞典（新人向け）

- **PoC**：本番前に小さく試して確かめること。
- **スナップショット**：ある時点の状態をまとめたデータ。
- **エスケープ**：文字を画面に安全に出すための変換。
- **モック**：本物の代わりに使う偽の部品（テスト用）。
- **査読済み論文**：専門家のチェックを通った信頼できる論文。
